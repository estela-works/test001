# 開発プロセスルール

## マネジメントチケットと開発フローの連携

### 開発タスクとマネジメントタスクの区別

#### 開発タスク
- `docs/projects/YYYYMMDD_案件名/` で管理
- 機能追加・バグ修正・リファクタリング
- コード変更を伴う作業

#### マネジメントタスク
- `management/tickets/` で管理
- プロセス改善・ルール変更・運用見直し
- ワークフロー・ドキュメント体系の変更

### マネジメント変更の適用タイミング

1. **即時適用**: ルール変更チケット作成時
   - チケット内 `rules/` を優先適用
   - 実験的な運用ルールを試行

2. **段階適用**: チケット完了時
   - 効果を確認後、グローバルルールに反映
   - `management/rules/current/` に統合

## マネジメントチケットの粒度

### 推奨サイズ

- **小規模**: 1週間以内に完了するもの
  - 例: ドキュメントテンプレート改善
  - 例: チケット命名規則の見直し

- **中規模**: 1〜2週間程度
  - 例: ワークフロー全体の見直し
  - 例: ドキュメント体系の再編成

- **大規模**: 1ヶ月程度
  - 複数の小規模チケットに分割を検討
  - マイルストーンを設定

## Git 運用との連携

### ブランチ戦略

マネジメントチケットの変更は基本的に `main` ブランチに直接コミット:
- ドキュメント・ルール変更は即座に共有
- コードに影響を与えないため、ブランチ分離不要

### コミットメッセージ規約

```
[Management] チケット名: 変更内容

例:
[Management] ワークフロールール追加: マネジメントチケット運用フローを定義
```

## レビュープロセス

### セルフレビュー

- チケット完了前に `notes.md` で振り返り
- ルール変更が矛盾していないか確認
- 既存ドキュメントとの整合性チェック

### チーム運用時のレビュー

- マネジメントルール変更は PR でレビュー
- 影響範囲が大きい場合はドラフト段階で相談
- 合意形成後に `rules/current/` に反映

## 継続的改善

### 定期見直しのタイミング

- 月次: マネジメントチケットの棚卸し
- 四半期: ルール全体の見直し
- 半期: アーカイブの整理

### メトリクス

- チケット消化率（Open vs Done）
- ルール変更頻度
- ドキュメント更新頻度
